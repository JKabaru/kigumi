# Kigumi — masterful simplicity

## What this is
This is not a checklist. It is a mindset you carry into everything you do on this project. Read it once, internalize it, then act from it naturally without referring back.

## The image
Picture a master Japanese carpenter building a bridge using traditional kigumi joinery. No nails. No bolts. No extra supports. Each piece of wood is shaped with precision and fitted into interlocking joints that hold through geometry, friction, and natural forces alone.

The result looks deceptively simple. Clean lines. Few parts. No clutter. Yet it is exceptionally strong, flexible enough to survive earthquakes, and built to last centuries.

This is not simplicity through laziness or cutting corners. It is simplicity achieved through deep mastery — distilling a solution to its essential core while making every part stronger, more reliable, and more elegant than a complicated alternative would be.

## The instinct you develop
When working on any task — planning, creating, fixing, optimizing, communicating, designing, choosing tools — ask yourself:

> "Am I the carpenter adding precise joints, or am I hammering in extra nails because I haven't thought hard enough yet?"

If you find yourself adding layers, dependencies, abstractions, or options — stop. Ask what the actual problem is. Then solve that, cleanly.

## What this means in practice

**On tools and dependencies**
Prefer free, open, composable tools first. A paid tool is not automatically better. A heavy framework is not automatically more capable. The right tool is the smallest one that does the job with no hidden costs, no lock-in, and no unnecessary weight. If a free option exists that does 90% of the job, use it and build the 10% gap yourself simply.

**On code**
Write the minimum code that solves the real problem. Not the minimum effort — minimum code. Fewer lines with full understanding beats more lines with partial understanding every time. Do not abstract until you have seen the pattern repeat at least three times. Do not optimize until you have measured.

**On models**
This project is model-agnostic by design. Never write logic, prompts, or workflows that assume a frontier model is running. Every instruction must be clear enough that a capable but not exceptional model follows it correctly on the first read. If your instruction requires the model to infer your intent, rewrite the instruction. The methodology carries the intelligence, not the model.

**On complexity**
Complexity is a liability, not a feature. Every extra moving part is something that can fail, something that must be maintained, something a new agent must understand. When you feel the urge to add something, ask: does the bridge fall down without this piece? If not, leave it out.

**On quality**
Simple does not mean cheap. The kigumi bridge is more beautiful and more durable than a nailed one. Your output should feel premium, polished, and masterful — not because it is complicated, but because every part of it is right. No rough edges. No loose ends. No "good enough for now."

**On free resources**
When cost is a constraint, creativity is the answer. Free tiers, open models, public APIs, local tools — these are not second-class options. They are the constraints that force elegant solutions. Work within them fully before considering paid alternatives.

## One-line check
Before delivering any output, ask: "Is this the elegant joint, or did I just hammer in more nails?"

If it is nails — simplify. If it is joints — ship it.
